# Part 4: Technical Communication

I chose the web readonly PR because it is the clearest example of a small, high-confidence change with a well-defined boundary. It introduces one config flag, one default policy, and a set of predictable route-level checks. Compared with the importer refactor, the replaygain parallelization work, or the flexible metadata changes, this PR is much easier to reason about because it does not change the library model, background processing, or query construction. The failure modes are also easy to test: either the route returns 405 or it performs the existing action.

My technical background fits this PR well because I’m comfortable with Python service code, HTTP request handling, configuration plumbing, and test-driven changes. I can trace how a flag moves from config into runtime behavior and how that affects route dispatch. I’m less comfortable with deep refactors that require understanding multiple layers of implicit model behavior, which is why I would not have picked the duplicate-key or flexible-tag work as my first choice.

The main implementation challenge I would expect is consistency. The read-only guard has to be applied in every place where DELETE and PATCH can mutate data, and it needs to stay aligned with both item and album routes. Another challenge is preserving backward compatibility for users who intentionally depend on writable web endpoints. I would handle that by keeping the default restrictive, documenting the override clearly, and verifying both modes with integration tests rather than relying on a single happy-path unit test.

If I were implementing this from scratch, I would start by tracing the current web route handlers, then wire the config value into the app object, and finally add tests for the blocked and allowed cases. I would specifically check that GET remains unaffected, that query-based deletes are blocked the same way as ID-based deletes, and that writable mode still behaves exactly like the pre-change code. That keeps the fix small, observable, and safe to merge.
